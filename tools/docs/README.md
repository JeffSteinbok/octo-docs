# Docs Generator

This directory contains the public documentation generation system for the octo-docs repository.

Documentation is generated by consuming a **sanitized bundle** produced by the private repo, running it through an LLM API, and writing polished Markdown pages.

This system does **not** access the private source repository directly.

---

## Architecture

```
Private Repo
   │
   │  1. uploads sanitized bundle as artifact
   │  2. sends repository_dispatch event (bundle-ready)
   ▼
octo-docs (this repo)
   │
   ├─ downloads bundle artifact
   ├─ loads page specs
   ├─ builds page prompts
   ├─ calls LLM API (GitHub Models via GITHUB_TOKEN)
   ├─ writes docs files
   └─ opens PR
   ▼
GitHub Pages
```

---

## Input Bundle Structure

The generator expects a bundle directory with this structure:

```
bundle/
  api-index.json
  modules/
    auth.json
    billing.json
  concepts/
    architecture.md
  release/
    changes.json
  manifest.json
  changed_pages.json
```

### `manifest.json`

```json
{
  "generated_at": "2026-03-07T20:10:00Z",
  "commit": "abc1234",
  "bundle_version": 7,
  "artifacts": ["modules/auth.json", "release/changes.json"]
}
```

### `changed_pages.json`

```json
{
  "pages": ["auth-overview", "release-latest"]
}
```

## Page Specs

Page specs live in `tools/docs/page_specs/*.yml` and define how each documentation page is generated.

Each spec includes:
- `id` — unique page identifier (must match `changed_pages.json` entries)
- `output_path` — where to write the generated file
- `sources` — which bundle files to include, and which fields to select
- `template` — which prompt template to use (`overview`, `api_reference`, `release_notes`)
- `instructions` — audience, include/exclude guidance for the LLM

### Example

```yaml
id: auth-overview
output_path: docs/auth/overview.md

sources:
  - path: modules/auth.json
    select:
      - summary
      - endpoints

template: overview

instructions:
  audience: external developers
  include:
    - overview
    - authentication flow
  exclude:
    - internal service names
```

Page specs are validated against `tools/docs/schemas/page_spec.schema.json`.

## Running Locally

### Prerequisites

```bash
pip install -r tools/docs/requirements.txt
```

Set LLM credentials:

In GitHub Actions the built-in `GITHUB_TOKEN` is automatically used with the
[GitHub Models](https://github.com/marketplace/models) OpenAI-compatible endpoint
(`https://models.inference.ai.azure.com`). No additional secrets configuration is
required. For local use, create a GitHub personal access token and export it:

```bash
export GITHUB_TOKEN=ghp_your-token
export DOCS_LLM_MODEL=gpt-4o            # optional, default: gpt-4o
```

### Generate changed pages only

```bash
python tools/docs/generation/generate_all.py --bundle ./bundle
```

### Regenerate all pages

```bash
python tools/docs/generation/generate_all.py --bundle ./bundle --all
```

### Regenerate a single page

```bash
python tools/docs/generation/generate_all.py --bundle ./bundle --page auth-overview
```

### Dry run (no LLM calls, no file writes)

```bash
python tools/docs/generation/generate_all.py --bundle ./bundle --dry-run
```

## How PR Creation Works

After generation, the gitops scripts create a branch, commit the changes, and open a PR:

```bash
python tools/docs/gitops/create_branch.py --bundle-root ./bundle
python tools/docs/gitops/commit_changes.py
python tools/docs/gitops/open_pr.py --bundle-root ./bundle
```

Requires:
- `GITHUB_TOKEN` — GitHub token with `contents: write` and `pull-requests: write`
- `GITHUB_REPOSITORY` — set to `owner/repo`

## Full Rebuild via Workflow

The workflow at `.github/workflows/generate-docs.yml` can be triggered in three ways:

### Automatic — repository_dispatch from the private repo

The private repo uploads the bundle as an artifact, then sends a `bundle-ready` dispatch event:

```bash
gh api repos/JeffSteinbok/octo-docs/dispatches \
  --method POST \
  -f event_type=bundle-ready \
  -F client_payload[artifact_name]=docs-bundle \
  -F client_payload[run_id]=$GITHUB_RUN_ID \
  -F client_payload[regenerate_all]=false
```

The workflow will download the artifact automatically and open a PR with the updated docs.

### Manual — download an artifact from a previous run

Trigger the workflow from the Actions tab and fill in:
- **Artifact name** — name of the artifact containing the bundle
- **Run ID** — the run that uploaded that artifact

### Manual — provide a bundle path directly

Leave `artifact_name` blank and set `bundle_path` to a bundle already on disk in the workspace.

## Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `GITHUB_TOKEN` | Yes | GitHub token — used for LLM calls (GitHub Models) and PR creation |
| `DOCS_LLM_MODEL` | No | Model name (default: `gpt-4o`) |
| `DOCS_LLM_BASE_URL` | No | Optional override for the GitHub Models base URL |
| `GITHUB_REPOSITORY` | Yes (for PRs) | `owner/repo` of this repository |
